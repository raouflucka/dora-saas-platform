[1] CHAPTER 4: SYSTEM ARCHITECTURE
[2] Chapter Overview
[3] This chapter presents the architectural design of the DORA SaaS platform — a multi-tenant web application developed to enable Irish SME financial entities to manage, validate, and export their DORA Register of Information (RoI) in compliance with the EBA's Implementing Technical Standards. The chapter is structured around eight core architectural concerns: the high-level layered overview; the backend module architecture; the data model and multi-tenancy design; the regulatory mapping of the data model to DORA Articles 28–30 and EBA templates RT.01–RT.09; the validation engine and its rule lifecycle; the RoI export subsystem; the security architecture; and the frontend architecture. Each section explains not only what the architecture does but why specific design choices were made, grounding those choices in established software engineering principles, regulatory requirements, and the constraints of an SME-facing prototype deployment. The chapter concludes with a brief summary that bridges into Chapter 5's implementation narrative.
[4] 4.1 Introduction to the System Architecture
[5] The DORA SaaS platform is designed to address a concrete and well-evidenced problem in the Irish financial sector: the inability of most SME financial entities to produce DORA-compliant Registers of Information that satisfy the EBA's machine-verifiable data quality standards. As established in Chapter 2, the ESA 2024 Dry Run found that only 6.5% of participating entities passed all validation checks, with the predominant failure causes being missing mandatory fields, invalid Legal Entity Identifiers, non-conforming DPM code values, and broken cross-template referential links (European Supervisory Authorities, 2024). These failures are not primarily attributable to ignorance of regulatory requirements but to the structural inadequacy of manual spreadsheet-based data management for a relational, machine-verifiable dataset of the complexity the RoI demands (European Banking Authority, 2023a; Galati et al., 2022).
[6] The architectural response developed in this dissertation is a purpose-built, multi-tenant SaaS system that operationalises the RoI as a live relational database, enforces data quality through an embedded declarative rule-based validation engine, and exports compliant outputs in both Excel and XBRL OIM-CSV formats for submission to the Central Bank of Ireland. The system is designed according to three overarching architectural principles derived from the literature review in Chapter 2 and the Design Science Research methodology in Chapter 3. First, regulatory traceability: every database table, field, and validation rule is explicitly mapped to the DORA article and EBA template column it serves, making the regulatory correspondence of the system verifiable rather than assumed (Palmirani et al., 2018; Bass et al., 2021). Second, transparency: all validation logic is stored as inspectable database records rather than embedded in opaque application code, implementing the white-box compliance model identified in Section 2.3 as a governance requirement for SME entities (Butler and O'Brien, 2019; Governatori et al., 2016). Third, proportionality: the system is sized and structured for resource-constrained SME organisations rather than for large financial groups, influencing technology selection, deployment model, and user interface design throughout (Bhurat, 2025; Central Bank of Ireland, 2024a).
[7] The architecture described in this chapter represents the system as implemented at the time of research — a fully functional prototype running in a local development environment, deployable via Docker Compose, and accessible through a web browser. It is not a production deployment and does not yet include cloud hosting, managed infrastructure, or formal penetration testing. These limitations are acknowledged explicitly throughout this chapter and addressed further in Chapter 5.

[9] 4.2 High-Level Architectural Overview
[10] 4.2.1 Layered Three-Tier Architecture
[11] The system adopts a three-tier layered architecture comprising a presentation tier (React SPA), an application tier (NestJS REST API), and a data tier (PostgreSQL relational database). This pattern is well-established for web-based information systems and provides clear separation of concerns between user interface rendering, business logic processing, and data persistence (Bass et al., 2021; Fowler, 2002). In regulatory compliance contexts, the layered pattern has the additional benefit of making the business and data logic independently testable and auditable — a property directly relevant to systems that must demonstrate compliance correctness (Palmirani et al., 2018; Bass et al., 2021).
[12] At the presentation tier, the frontend is a single-page application (SPA) built with React 19, Vite, TypeScript, TailwindCSS, and Shadcn UI. It communicates exclusively with the application tier through versioned REST JSON endpoints. At the application tier, the NestJS backend provides a modular structured monolith organised into domain-specific modules, each encapsulating its own controller, service, and Prisma data-access logic. At the data tier, a single PostgreSQL instance hosts the shared-schema multi-tenant database, with all domain tables carrying a tenant_id foreign key that the application tier enforces at the service layer, additionally reinforced at the database engine layer through PostgreSQL Row-Level Security policies. Newman (2019) describes this "modular monolith" pattern as the appropriate starting point for domain-driven systems where clear internal module boundaries are required but microservices operational overhead is disproportionate to the deployment scale — a characterisation that precisely fits the present research context.
[13] The system also includes three cross-cutting components that are not tied to a single tier. The MailerModule (nodemailer 8.0.4) handles transactional email delivery for password reset flows. The NestJS ThrottlerModule (throttler 6.5.0) enforces rate limiting across all API endpoints. The AuditInterceptor — a globally registered NestJS interceptor — captures before-and-after state for all mutating API operations, writing entries to the audit_logs table regardless of which module's controller handled the request.
[14] 4.2.2 Technology Selection
[15] The technology stack is summarised in Table 4.1 below. The backend is built on NestJS — a TypeScript-native, module-structured Node.js framework that enforces separation of concerns through its decorator-based controller/service/module pattern and provides first-class integration with dependency injection, guards, interceptors, and pipes (NestJS, 2023). NestJS is a widely adopted choice for enterprise-grade REST APIs in the Node.js ecosystem and is well-documented in the academic and practitioner literature on TypeScript backend architecture (Gerasimov, 2019). Prisma (version 7.5.0) serves as the ORM and schema management layer, providing type-safe database access and a declarative schema definition that drives database synchronisation via prisma db push during development (Prisma, 2024). PostgreSQL 16 provides the relational storage engine, chosen for its maturity, ACID transaction guarantees, native JSONB support (used for the old_values and new_values fields in audit_logs), and native Row-Level Security policies that enforce tenant isolation at the database engine layer (PostgreSQL Global Development Group, 2023).
[16] The frontend stack is built around React 19 with the Vite build tool and TypeScript, providing a component-based SPA architecture with full static type coverage. TailwindCSS and Shadcn UI provide the design system; Zustand manages global authentication and UI state; and TanStack Query (React Query v5) manages all server-state fetching, caching, mutation, and optimistic update logic. React Router v6 provides client-side routing with role-based route guards implemented as HOC wrappers that check the Zustand authentication store before rendering protected routes.

[18] 4.3 Backend Architecture and Services
[19] 4.3.1 Module Structure and Design Principles
[20] The NestJS backend is organised as a structured monolith comprising twenty-one domain and infrastructure modules, all registered in a root AppModule. Each module follows NestJS's standard pattern: a Module decorator declares the module's imports, controllers, providers, and exports; a Controller handles HTTP routing and request/response transformation; a Service contains business logic and Prisma data-access calls; and — where applicable — DTO classes annotated with class-validator decorators enforce input validation at the controller boundary. This consistent structure implements the layered responsibility pattern described by Fowler (2002), ensuring that HTTP concerns, business logic, and data access are cleanly separated and independently testable.
[21] The module boundaries are deliberately aligned with DORA's own conceptual structure: each of the nine RoI template domains has a corresponding backend module, supplemented by infrastructure modules for authentication, user management, audit logging, reference data, notifications, and export. This regulatory-domain alignment is consistent with the recommendation in Bass et al. (2021) that system architecture should reflect the domain structure it serves, both to improve maintainability and to make the correspondence between system design and regulatory requirements visible to reviewers and auditors.
[22] 4.3.2 Domain Modules
[23] The platform's domain modules are structured to map directly onto the RoI template hierarchy defined by the EBA's ITS. Each module owns the CRUD endpoints and business logic for its corresponding database tables:
[24] FinancialEntitiesModule (/api/v1/financial-entities, /api/v1/branches) — manages the financial_entities and branches tables, which together constitute the RT.01 data. The module handles creation, retrieval, update, and deletion of entity records with full tenant scoping, and supports branch-to-entity linking through a branchOf foreign key relationship.
[25] IctProvidersModule (/api/v1/ict-providers) — manages the ict_providers table (RT.03), storing provider identity, LEI, NACE code, country of establishment, and type classification.
[26] ContractualArrangementsModule (/api/v1/contractual-arrangements) — the contract hub of the system, managing the contractual_arrangements table (RT.02) and its three linking tables. The ContractEntitiesModule (/api/v1/contract-entities) records which financial entities are party to each contract; the ContractProvidersModule (/api/v1/contract-providers) records which providers are signatories. A third linking table, entities_using_services, captures which entity uses which service under which contract.
[27] FunctionsModule (/api/v1/functions) — manages business_functions and the function_ict_dependencies linking table (RT.04), storing function identifiers, criticality classifications, and the contract-to-function dependency relationships.
[28] IctServicesModule (/api/v1/ict-services) — manages the ict_services table, representing ICT services as discrete assets associated with providers, functioning as an internal named-service asset register analogous to a CMDB entry.
[29] IctServiceAssessmentsModule (/api/v1/assessments) — manages the ict_service_assessments table (RT.07), storing substitutability assessments, data sensitivity levels, and reliance levels for each ICT service assessment record.
[30] IctSupplyChainModule (/api/v1/ict-supply-chain) — manages the ict_supply_chain table (RT.05.02), recording supply chain entries linking contracts, providers, subcontractors, service types, and supply rank. Supply chain CRUD is available to both ADMIN and EDITOR roles.
[31] ExitStrategiesModule (/api/v1/exit-strategies) — manages the exit_strategies table (RT.08), storing contractual exit provisions, exit triggers, strategy narratives, and fallback provider linkage.
[32] 4.3.3 Infrastructure Modules
[33] Alongside the domain modules, five infrastructure modules provide platform-wide services:
[34] AuthModule (/api/v1/auth) — handles login, logout, token refresh, password reset, and the JWT/Passport authentication strategy. This module issues access tokens (15-minute TTL) and refresh tokens (7-day TTL), implementing per-use token rotation as described in Section 4.8.
[35] UsersModule (/api/v1/users) — handles user management by ADMINs and user self-service (profile update, password change) via PATCH /api/v1/users/me/profile and PATCH /api/v1/users/me/password.
[36] ValidationModule (/api/v1/validation) — contains the complete rule-based engine, manages validation_runs and validation_issues, and exposes the pre-flight gate used by the export subsystem.
[37] RoiExportModule (/api/v1/roi) — handles Excel and XBRL OIM-CSV export generation, using ExcelJS (4.4.0) and the Node.js archiver library for ZIP packaging.
[38] ReferenceModule (/api/v1/reference) — provides read-only endpoints for all nine lookup/reference tables, enabling the frontend to populate compliant dropdown lists from server-authoritative EBA code lists.
[39] Additional supporting modules — AuditLogModule, NotificationsModule, CommentsModule, DashboardModule, AdminModule, TenantsModule, and MailerModule — provide the audit trail, in-app notification system, collaborative comment threads on validation issues, dashboard aggregate statistics, administrative user and tenant management, and password-reset email delivery respectively. The API surface is versioned uniformly under /api/v1/ and is fully documented through a Swagger/OpenAPI interface accessible at /api/docs.

[41] 4.4 Data Model and Multi-Tenancy Design
[42] 4.4.1 The Shared-Schema Multi-Tenancy Model
[43] The platform implements a shared-schema, application-enforced multi-tenancy model, in which all financial entities — acting as tenants — share a single PostgreSQL database schema and a single set of tables, with tenant isolation enforced by a tenant_id UUID foreign key present on every domain table. This model is consistent with the practical multi-tenancy approach described by Bezemer et al. (2010) and Chong and Carraro (2006) for SaaS applications where operational simplicity is prioritised and where the number of tenants is moderate. The economics of this model are well-suited to the present context: a single database instance, a single schema definition, and a single deployed backend serve all tenants, reducing infrastructure complexity and operational cost proportionally.
[44] Tenant isolation is enforced through a defence-in-depth strategy operating at two independent layers. At the application service layer, the TenantIsolationMiddleware — registered globally in AppModule — fires on every request, extracts the tenantId claim from the validated JWT payload, and sets the app.current_tenant_id PostgreSQL session variable. Every Prisma query in every service method appends a mandatory where: { tenantId: request.user.tenantId } clause, derived from the token rather than from any user-submitted parameter. At the database engine layer, PostgreSQL Row-Level Security (RLS) policies are applied to all 20 tenant-bearing tables via rls_policies.sql, which activate using the session variable set by the middleware on every request (PostgreSQL Global Development Group, 2023). This dual-layer approach ensures that even if an application-layer filtering bug were to occur, the database engine independently enforces tenant boundaries, consistent with the defence-in-depth architecture described by Rodero-Merino et al. (2010).
[45] The tenants table records the identity and configuration of each tenant, with a UUID primary key (id) that appears as tenant_id on all domain tables. Each users record is associated with exactly one tenant through a tenant_id foreign key, constraining user scope to their organisation's data. ADMIN-role users can manage users and configuration within their own tenant; tenant provisioning is a manual onboarding step in the current prototype.
[46] 4.4.2 Table Groups and the RoI Data Chain
[47] The database schema is organised into three conceptual groups: reference tables, system and security tables, and DORA domain tables.
[48] Reference tables store the EBA's standardised code lists, which are consumed by the application to constrain categorical field values to permitted entries. Nine reference tables are defined: countries (ISO 3166-1 alpha-2), currencies (ISO 4217), entity_types (EBA DPM entity type codes), criticality_levels, reliance_levels, data_sensitivity_levels, ict_service_types, provider_person_types, and user_roles. These tables are seeded at application initialisation from the EBA's published DPM code lists and are read-only at runtime (European Banking Authority, 2023a).
[49] System and security tables include tenants, users, audit_logs, notifications, comments, validation_rules, validation_runs, and validation_issues. The users table stores hashed passwords (bcrypt, ten rounds), JWT-related fields (refresh_token_hash, reset_token, reset_token_expires), role assignment, and tenant association. The audit_logs table stores JSONB old_values and new_values snapshots for every mutating operation, alongside user identity, tenant identity, target table name, record ID, and HTTP method. The validation_rules, validation_runs, and validation_issues tables constitute the validation engine's persistent state, described in detail in Section 4.5.
[50] DORA domain tables encode the nine-template RoI dependency chain as a normalised relational schema. The chain follows the regulatory logic defined in DORA Articles 28–30 and the EBA ITS: financial_entities → branches → business_functions → function_ict_dependencies → contractual_arrangements → ict_providers → ict_services → ict_supply_chain → ict_service_assessments → exit_strategies. Three linking tables — contract_entities, contract_providers, and entities_using_services — model the many-to-many relationships between contracts and their parties as required by the RoI template structure (European Banking Authority, 2023a). Every domain table carries tenant_id, created_at, updated_at, and a UUID id primary key.

[52] 4.4.3 Referential Integrity and Schema Constraints
[53] Prisma's @relation declarations define all foreign key relationships at the schema layer, which Prisma translates to database-level FOREIGN KEY constraints. This means that referential integrity — for example, the requirement that a contractual_arrangements.provider_id value must exist in ict_providers — is enforced at the database engine level, independent of application logic. This two-layer approach — application-level validation plus database-level constraint — is consistent with best practice for financial data systems, where data corruption through partial updates or bypassed service layers must be prevented (Fehling et al., 2014; Bass et al., 2021). The EBA's cross-template integrity rules, which require that identifiers referenced in one template appear in the appropriate source template, are modelled through this same foreign-key mechanism wherever the data model allows direct table-to-table referencing.
[54] 4.5 Regulatory Mapping: DORA Articles and EBA Templates
[55] 4.5.1 Mapping to DORA Articles 28–30
[56] DORA Articles 28 through 30 define the substantive regulatory obligations that the RoI is designed to document. Article 28 requires financial entities to maintain registers of all contractual arrangements with ICT third-party service providers, with particular detail required for arrangements supporting critical or important functions. Article 29 addresses the management of ICT concentration risk arising from the use of a limited number of critical ICT third-party service providers. Article 30 specifies the key contractual provisions that must be included in agreements with ICT service providers (European Commission, 2022, Arts. 28–30).
[57] The DORA SaaS data model maps to these articles as follows. Article 28's contractual arrangement inventory is implemented through the contractual_arrangements table (RT.02), which records each contract's unique reference, the contracting parties, the nature of the service, the governing law, and the start and end dates. The mapping of critical and important functions is implemented through business_functions with a criticality field constrained to the EBA's criticality code list (RT.06.01), and through function_ict_dependencies which links each function to the contracts that support it (European Commission, 2022, Art. 28(2)). Article 29's concentration risk visibility is supported through the ict_supply_chain table (RT.05.02), which records sub-contractor relationships below the primary provider tier, enabling the cross-tenant aggregation that supervisors need to detect systemic concentration (European Systemic Risk Board, 2022). Article 30's contractual provisions are partially modelled through exit_strategies (RT.08), which records exit plan timelines and substitution feasibility assessments.

[70] 4.6 Validation Engine and Rule Lifecycle
[71] 4.6.1 Architectural Design of the Validation Engine
[72] The validation engine is the architectural component most directly instantiating the rule-based compliance approach described in Chapter 2. Its design embodies the declarative, data-driven pattern recommended by Sadiq et al. (2007) and Kharbili et al. (2008): all validation rules are stored as structured records in the validation_rules database table rather than embedded in application code. The execution engine in ValidationService reads the active rules from the database at run time, applies each rule's logic generically to the target dataset, and produces a structured list of issues. This separation between rule definition (database) and rule execution (application code) means that regulatory rule updates — which are expected to occur as supervisory experience with DORA accumulates — can be applied through a database seeder re-run rather than through an application code change and deployment (Governatori et al., 2016; Sadiq et al., 2007).
[73] The validation_rules table captures the following fields for each rule: id (UUID primary key), templateName (e.g. "RT.02", "RT.07.01"), fieldName (the specific EBA column or platform field the rule applies to), ruleType (one of the ten supported rule type codes), ruleValue (the constraint parameter — a regex pattern, a foreign key table name, a min/max range, or a conditional expression), errorMessage (human-readable message displayed to the user), severity (ERROR, WARNING, or INFO), doraArticle (regulatory citation e.g. "Art.28(1)"), and isActive (boolean allowing rules to be disabled without deletion). This schema is directly informed by the EBA's published draft validation rules specification (European Banking Authority, 2024a), which classifies rules by template, field, rule type, and severity.
[74] 4.6.2 Rule Types and Coverage
[75] The engine supports ten rule types, each corresponding to a distinct class of EBA validation requirement:
[76] required — validates that a mandatory field contains a non-null, non-empty value. Example: financial_entities.lei must be present for all RT.01 records. This type covers the single largest failure category identified in the Dry Run (European Supervisory Authorities, 2024).
[77] format — validates that a field value matches a defined regular expression. Example: LEI values must be 20-character alphanumeric strings conforming to ISO 17442 (ISO, 2020); date fields must match YYYY-MM-DD format.
[78] fk_exists — validates referential integrity across templates: the value in a foreign-key field must correspond to an existing record in the referenced table within the same tenant. Example: function_ict_dependencies.contract_id must reference an existing record in contractual_arrangements for the same tenant_id. This rule type directly enforces the cross-template integrity rules that are a primary source of Dry Run failures (European Supervisory Authorities, 2024; European Banking Authority, 2024a).
[79] range — validates that a numeric value falls within defined minimum and maximum bounds. Example: supply_rank must be a positive integer within the permitted sub-contracting depth.
[80] dropdown — validates that a field value is one of the permitted entries in the corresponding EBA code list reference table. Example: entity_type must be one of the values in the entity_types reference table.
[81] cross_field — validates a condition involving two or more fields within the same record. Example: end_date must not be earlier than start_date on a contractual_arrangements record.
[82] conditional — validates a field only when a condition on another field is true. Example: substitutionReason is required when isSubstitutable is false on ict_service_assessments records (EBA VR109).
[83] date_boundary — a specialised date constraint validating that a date field does not fall before a defined regulatory boundary. Example: contract start dates must not precede the DORA application date of 17 January 2025 for new arrangements.
[84] uniqueness — validates that a field value is unique within the tenant's dataset. Example: financial_entities.lei must be unique per tenant.
[85] aggregate — validates cross-record conditions such as concentration thresholds or minimum counts. Example: verifying that a concentration risk threshold is not exceeded for a given provider across all contracts.
[86] The engine seeds 220 EBA validation rules across templates RT.01–RT.09, drawn directly from the EBA's published draft validation rules spreadsheet (EBA VR_01–VR_250). These 220 rules represent all ten rule types and provide full mandatory-field coverage for templates RT.01 through RT.08, with partial coverage for RT.09. The complete EBA draft validation rules spreadsheet contains approximately 300 rules; the 73% implementation coverage defers advanced inter-template cross-checks and certain advanced RT.09 concentration risk rules to future development (European Banking Authority, 2024a).
[87] 4.6.3 Validation Run Lifecycle and State Machine
[88] A validation run is initiated by a POST request to /api/v1/validation/run from an authenticated ANALYST user. The ValidationService retrieves all active rules from validation_rules, iterates through each rule in execution sequence for the relevant templates, queries the applicable domain table filtered by tenant_id, and evaluates each rule's condition against each record. Each failure produces a validation_issues record capturing the rule ID, the record ID and table, the specific field, the error message, the EBA rule identifier, the DORA article reference, and the issue severity.
[89] The validation_issues record follows a five-state lifecycle that governs the collaborative remediation workflow:
[90] OPEN — newly created by a validation run; requires attention.
[91] FLAGGED — marked by an ANALYST as requiring Editor remediation; a notification is sent to the EDITOR role.
[92] WAITING_APPROVAL — an EDITOR has applied a fix to the underlying data and submitted for Analyst review.
[93] RESOLVED — an ANALYST has confirmed the fix; the issue is permanently excluded from future run results.
[94] FIXED — the issue no longer fires on a subsequent validation run after the underlying data was corrected; auto-closed by the engine.
[95] This role-differentiated state machine implements the collaborative compliance workflow described in the RegTech literature: separation of the flagging role (Analyst) from the remediation role (Editor) from the approval role (Analyst/Admin) distributes the compliance workload across the user population while maintaining a clear audit trail of who acted at each stage (Butler and O'Brien, 2019; Arner et al., 2017). Every state transition is written to the audit_logs table with the acting user, timestamp, and old/new state.
[96] The validation engine is directly integrated with the RoI export subsystem through a pre-flight gate: when an export request is received by RoiExportModule, the service first queries for any open ERROR-severity issues in the most recent validation run. If any such issues exist, the export request returns a structured error response identifying the blocking issues rather than generating the export file. This gate ensures that the platform cannot produce a RoI export containing data that fails EBA validation requirements, directly addressing the correctness quality attribute identified in Section 2.4 (Bass et al., 2021; European Banking Authority, 2024a).

[98] 4.7 RoI Export Subsystem
[99] 4.7.1 Dual-Path Export Architecture
[100] The RoI export subsystem implements a dual-path architecture providing two distinct output formats from a single underlying dataset: Excel-format exports that conform to the EBA's published template workbook structure, and XBRL OIM-CSV exports that conform to the EBA's machine-readable submission format for digital regulatory reporting. Both paths share the same data retrieval logic — a series of Prisma queries scoped by tenant_id that assemble the thirteen sub-template dataset from the domain tables — and diverge only in the serialisation and packaging step (European Banking Authority, 2023a; European Banking Authority, 2024b).
[101] The dual-path design reflects a deliberate accommodation of the CBI's current submission practice. At the time of research, the CBI was accepting both Excel and XBRL OIM-CSV submissions during the transition period, with the expectation that XBRL OIM-CSV would become the standard submission format as supervisory infrastructure matures (Central Bank of Ireland, 2024a). By implementing both paths, the platform supports entities at different stages of adoption without requiring them to manage format conversion externally.
[102] 4.7.2 Excel Export
[103] Excel export is implemented using ExcelJS (version 4.4.0), which provides a Node.js API for programmatic workbook creation with support for cell formatting, column widths, sheet naming, and data type annotation. The RoiExportService constructs a multi-sheet workbook with one worksheet per RoI sub-template — RT.01.01 through RT.09.01 — across all thirteen sub-templates. Each worksheet uses a two-row header structure with EBA column codes in the first row and human-readable labels in the second, exactly matching the EBA's published template workbook convention (European Banking Authority, 2023a).
[104] The endpoint GET /api/v1/roi/export triggers the generation, sets the Content-Disposition: attachment response header, and streams the workbook directly to the HTTP response. File naming follows a consistent convention using the tenant identifier and the generation date.
[105] 4.7.3 XBRL OIM-CSV Export
[106] The XBRL OIM-CSV export produces a ZIP archive containing one CSV file per RoI sub-template plus a metadata.json file. The archive structure follows the EBA's OIM-CSV specification: each CSV file is named according to the EBA convention; the first two rows contain EBA-defined column headers; and each subsequent row represents one record from the corresponding domain table, with null values rendered as empty fields as required by the OIM-CSV schema (European Banking Authority, 2024b).
[107] The metadata.json file records the submission date, the exporting entity's LEI, the reference period, and a schemaRef URI pointing to the EBA's published OIM-CSV taxonomy. The current implementation uses a static schemaRef value pointing to the most recently published EBA taxonomy version at the time of development. This is a known limitation: if the EBA publishes a revised taxonomy, the schemaRef will require a manual update in the export configuration.
[108] The endpoint GET /api/v1/roi/export/xbrl generates the ZIP archive, sets appropriate application/zip content-type headers, and streams the archive to the HTTP response using Node.js's archiver library. The pre-flight validation gate described in Section 4.6.3 applies equally to XBRL exports as to Excel exports, ensuring that both output paths are subject to the same data quality gate.
[109] 4.8 Security Architecture and Controls
[110] 4.8.1 Authentication
[111] Authentication follows a JWT-based, stateless design using @nestjs/passport with the passport-jwt strategy. When a user authenticates through POST /api/v1/auth/login, the AuthService verifies the submitted credentials against the bcrypt hash stored in users.passwordHash (ten salt rounds). On success, two tokens are issued: a short-lived access token (15-minute TTL) and a long-lived refresh token (7-day TTL). The access token's JWT payload encodes { id, email, tenantId, role }, providing the application tier with all necessary identity and authorisation context without requiring a database lookup on each request (NIST, 2018; ISO, 2022).
[112] Refresh tokens are 64-byte cryptographically random values generated using Node.js's crypto.randomBytes. The raw token value is transmitted to the client but only the bcrypt hash is persisted in users.refresh_token_hash. Per-use token rotation is implemented: each successful refresh call overwrites the stored hash with the new token's hash, invalidating the previous refresh token immediately. Both access and refresh tokens are delivered as HttpOnly cookies — the access token cookie is sent to all API paths; the refresh cookie is scoped to /api/v1/auth only, preventing it from being included in data-plane requests (ISO, 2022; Anderson, 2020).
[113] Token revocation is handled at logout (hash set to null), at password reset (hash cleared along with reset token fields), and on detection of refresh token theft (an invalid raw token presented for a valid user identity clears the hash, terminating all sessions). This multi-point revocation strategy is consistent with OWASP's recommendations for refresh token lifecycle management (OWASP, 2021) and NIST SP 800-63B's guidance on session management (NIST, 2018).
[114] 4.8.2 Authorisation and Role-Based Access Control
[115] The system defines three user roles — ADMIN, ANALYST, and EDITOR — enforced through a NestJS RolesGuard that is applied globally and reads a @Roles(...) decorator on each controller endpoint. ADMINs have full create-read-update-delete permissions across all domain data and administrative functions within their tenant. EDITORs can create, update, and delete domain data records (including supply chain entries) but cannot manage users, run validation, or access administrative views. ANALYSTs can view all data and manage validation runs and issue states but cannot create or modify domain data records.

[117] Role checks are enforced at two layers: the RolesGuard rejects requests from users whose JWT role claim does not match the @Roles(...) requirement before the request reaches the controller handler, providing a uniform role enforcement point independent of individual service method logic. Service methods additionally re-check request.user.tenantId for all data-plane operations, providing defence-in-depth against any hypothetical bypass of the guard layer (Bass et al., 2021; Anderson, 2020).
[118] 4.8.3 Additional Security Controls
[119] Rate limiting is enforced through NestJS's ThrottlerModule (version 6.5.0) with a global limit of 10 requests per 60 seconds per IP address. This configuration mitigates brute-force and enumeration attacks against the API without creating unacceptable friction for legitimate users (OWASP, 2021).
[120] Security headers are applied globally through the helmet middleware (version 8.1.0), which sets Strict-Transport-Security, X-Frame-Options: DENY, X-Content-Type-Options: nosniff, Content-Security-Policy, and Referrer-Policy headers on all responses. CORS is configured to allow requests only from the expected frontend origin (http://localhost:8000), with credentials (cookies) permitted only to authenticated endpoints (Anderson, 2020; OWASP, 2021).
[121] Input validation is performed at the controller boundary using class-validator DTOs: every POST and PATCH request body is parsed against a DTO class with field-level validation annotations (@IsString(), @IsUUID(), @IsDateString(), @IsEnum(), etc.) before reaching the service layer. The ValidationPipe is configured with whitelist: true and forbidNonWhitelisted: true, ensuring that undeclared fields are stripped and rejected respectively.
[122] Audit logging is implemented by the globally registered AuditInterceptor. For every mutating request (POST, PATCH, PUT, DELETE), the interceptor pre-fetches the current record state through AuditFetchService before forwarding to the handler, then records the old_values (pre-mutation) and new_values (post-mutation) as JSONB objects in audit_logs, alongside the user ID, tenant ID, target table, record ID, HTTP method, and timestamp. The sanitize() method strips credential-related fields (passwordHash, resetToken, refreshTokenHash) before persistence. All audit log entries are available to ADMIN-role users through the /admin/audit-logs frontend page.

[124] 4.9 Frontend Architecture and User Experience
[125] 4.9.1 SPA Architecture and State Management
[126] The frontend is a single-page application (SPA) built with React 19, Vite, and TypeScript. The SPA architecture provides a responsive, application-like user experience without full page reloads between navigation events, which is important for a compliance management tool where users frequently move between related data domains — for example, from a contract record to its associated provider and then to its validation issues (Garrett, 2005). Vite provides a fast development server and optimised production build pipeline with tree-shaking and code splitting, keeping the initial bundle size manageable for users on standard enterprise connections.
[127] Global state is divided between two complementary stores. Zustand manages client-side persistent state — primarily the authenticated user object (id, email, role, tenantId), the active session status, and UI-level settings — because this state must survive component re-renders and must be synchronously accessible for route guard evaluation. TanStack Query (React Query v5) manages all server state — the fetched datasets returned by API calls — providing automatic caching, background refetching, stale-while-revalidate behaviour, and mutation lifecycle hooks. This division of state responsibilities is consistent with the recommended pattern in the React community for applications with significant data-fetching requirements (Tanner Linsley, 2023).
[128] 4.9.2 Route Structure and Role-Based Access
[129] Client-side routing is implemented with React Router v6. Protected routes are wrapped in a ProtectedRoute higher-order component that reads the Zustand authentication store and redirects unauthenticated users to /login. Role-specific routes are wrapped in a RoleGuard component that checks whether the authenticated user's role is included in the permitted roles array for that route. The route structure maps directly onto the DORA RoI template hierarchy and the platform's functional areas:
[130] / — Dashboard: aggregate statistics (entity count, provider count, outstanding validation issues, recent audit events)
[131] /entities — Financial entities and branches (RT.01)
[132] /providers — ICT providers (RT.05.01)
[133] /contracts — Contractual arrangements (RT.02) with contract-entities and contract-providers sub-sections
[134] /functions — Business functions and ICT dependencies (RT.06.01)
[135] /assessments — ICT service assessments (RT.07.01)
[136] /supply-chain — ICT supply chain (RT.05.02)
[137] /ict-services — ICT services (internal asset register)
[138] /exit-strategies — Exit strategies (RT.08.01)
[139] /validation — Validation dashboard: trigger runs, view issues by template and severity, manage issue state machine
[140] /roi-export — Export dashboard: template coverage grid, Excel/XBRL toggle, pre-flight status indicators, download triggers
[141] /admin — Administrative views: tenant management, user management, audit log viewer, concentration risk summary; ADMIN only
[142] /settings — User self-service: profile update and password change; all roles
[143] 4.9.3 UX Design for SME Compliance Users
[144] The user interface design is specifically oriented towards the characteristics of SME compliance users identified in the literature: non-specialist data entrants who need clear guidance on what values are permitted, what validation rules apply, and what actions are required to remediate issues (Butler and O'Brien, 2019; Bhurat, 2025). Three UX patterns are implemented consistently across all data entry forms.
[145] First, constrained dropdowns: every categorical field is rendered as a <Select> component populated from the ReferenceModule API, constraining input to EBA DPM code-list values before submission. This eliminates the "invalid code value" failure class that accounted for a significant proportion of Dry Run errors. Second, inline validation feedback: class-validator errors from the backend are surfaced as per-field error messages inline on the form, directing the user to the specific field requiring attention. Third, empty-state guidance: pages that have not yet had data entered display descriptive empty-state messages explaining what data is required and why, linking to the relevant DORA article where space permits. These patterns reflect the principle of compliance-by-design — structuring the user experience to make compliant data entry the path of least resistance (Palmirani et al., 2018; Butler and O'Brien, 2019).
[146] 4.10 Chapter Summary
[147] This chapter has described the complete system architecture of the DORA SaaS platform across its eight principal components. The three-tier layered architecture — React 19 SPA, NestJS structured monolith, PostgreSQL 16 — provides clear separation of concerns and maps domain structure directly to the DORA regulatory framework. The shared-schema multi-tenancy model delivers operational simplicity appropriate to the SME deployment context, with tenant isolation enforced through a defence-in-depth strategy combining application-layer tenant_id filtering with PostgreSQL Row-Level Security at the database engine layer across all 20 tenant-bearing tables. The data model's nine-template chain faithfully encodes DORA's RoI data structure as a normalised relational schema, with referential integrity enforced at the database level. The declarative rule-based validation engine — storing 220 EBA rules as database records interpreted by a generic ten-type execution engine — implements the white-box, rule-transparent compliance design identified in Chapter 2 as a governance requirement for SME entities. The dual-path export subsystem produces both Excel and XBRL OIM-CSV outputs through a pre-flight validation gate, ensuring that only quality-checked data reaches the regulatory submission formats. The security architecture applies a defence-in-depth strategy — JWT authentication with rotating refresh tokens, RBAC across three roles, dual-layer tenant isolation, rate limiting, security headers, and full audit logging with before-and-after value capture — across all API paths. The frontend's SPA architecture and SME-oriented UX patterns make constrained, compliant data entry the default experience.
[148] Chapter 5 now proceeds to describe how each of these architectural choices was concretely implemented, examining the specific modules, services, seeding strategies, and remaining limitations that characterise the artefact as built.
