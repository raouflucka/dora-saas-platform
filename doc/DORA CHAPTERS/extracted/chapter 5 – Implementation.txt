[1] CHAPTER 5: IMPLEMENTATION
[2] Chapter Overview
[3] This chapter describes the concrete implementation of the DORA SaaS platform, building directly on the architectural design established in Chapter 4. Where Chapter 4 explained what the system does and why it is structured as it is, this chapter explains how each component was built: the specific modules, services, data-access patterns, seeding strategies, authentication mechanics, export pipeline, and frontend flows that together instantiate the rule-based compliance framework proposed in the research objectives. The chapter proceeds through eight implementation domains — backend application layer, database implementation, security and access control, validation engine, RoI export pipeline, frontend implementation, testing and evaluation, and known limitations — and closes with a candid assessment of the gap between the prototype as built and a fully hardened production deployment. Throughout, implementation decisions are connected to their architectural motivations from Chapter 4 and, where appropriate, to the Design Science Research methodology introduced in Chapter 3, demonstrating that the artefact embodies the conceptual model it was designed to instantiate (Hevner et al., 2004; Peffers et al., 2007).
[4] 5.1 Implementation Objectives and Constraints
[5] 5.1.1 Guiding Principles
[6] Three principles guided implementation choices throughout the development of the DORA SaaS prototype. The first is regulatory fidelity: every implementation decision that touches the data model, the validation logic, or the export format was evaluated against the EBA's published ITS and the CBI's RoI submission guidance, with fidelity to the regulatory source taking precedence over implementation convenience (European Banking Authority, 2023a; Central Bank of Ireland, 2024a). This principle manifests in choices such as naming database columns to match EBA field names directly, seeding reference tables from the EBA's DPM code lists rather than deriving custom enumerations, and structuring XBRL OIM-CSV files according to the EBA's published CSV template conventions.
[7] The second principle is maintainability by design: given that DORA is a regulation whose technical standards are expected to evolve — the EBA itself acknowledged during consultation that the ITS would be subject to revision — the implementation prioritises configurations that are easy to update over configurations that require code changes (European Banking Authority, 2023b). This principle drives the decision to store all validation rules as database records rather than as hard-coded logic, and to manage reference data through seed scripts rather than embedded constants, so that regulatory amendments can be accommodated without touching the application tier.
[8] The third principle is honest proportionality: the system is a research prototype targeting SME entities in a local development context, not a production SaaS offering (Bhurat, 2025). Implementation choices reflect this scope honestly — using prisma db push rather than versioned migrations during iterative development, using Docker Compose rather than Kubernetes for orchestration, and acknowledging explicitly where automated test coverage is limited (Bass et al., 2021). This principle is reflected in the limitations discussion in Section 5.8.
[9] 5.1.2 Technology Versions
[10] The precise technology versions in the implemented prototype are: NestJS 11, Node.js 22, TypeScript 5.9, Prisma 7.5.0, PostgreSQL 16, React 19, Vite 6, TailwindCSS 3.4, Shadcn UI (component library built on Radix UI primitives), Zustand 5.0, TanStack Query 5.67.2, React Router 6, ExcelJS 4.4.0, @nestjs/jwt 11.0, bcrypt 5.1.1, helmet 8.1.0, and @nestjs/throttler 6.5.0. [14] 5.2.1 NestJS Module Structure in Practice (continued)
[15] Every controller is decorated with @UseGuards(JwtAuthGuard, RolesGuard) at the class level, which means that authentication and role enforcement are applied to all endpoints uniformly without requiring per-endpoint guard declarations. Any endpoint that must be publicly accessible — such as POST /auth/login and POST /auth/forgot-password — is explicitly decorated with @Public(), a custom metadata marker that causes JwtAuthGuard to skip token verification for that route only (Anderson, 2020).
[16] Request bodies are validated at the controller boundary through NestJS's built-in ValidationPipe, registered globally in main.ts with whitelist: true and forbidNonWhitelisted: true. These settings mean that any property in a request body that does not appear in the corresponding DTO class is silently stripped before the controller handler is invoked, and any request containing unrecognised properties returns a 400 error. This implements defence-in-depth at the API boundary: even if a service method were to inadvertently pass all properties of a request body to a Prisma create() or update() call, the stripped DTO guarantees that only declared fields reach the database (OWASP, 2021).
[17] Error handling is centralised through a global HttpExceptionFilter that intercepts all thrown exceptions — both NestJS HttpException instances and unexpected runtime errors — and converts them to a uniform JSON error envelope: { statusCode, message, error, timestamp, path }. This means that API consumers, including the frontend, always receive errors in a predictable structure and that stack traces and internal diagnostic messages are never exposed in production responses (Anderson, 2020; Bass et al., 2021).
[18] 5.2.2 The ContractualArrangements Hub
[19] The ContractualArrangementsModule required the most complex implementation because it serves as the relational hub connecting financial entities, ICT providers, business functions, ICT services, and supply chain records — the majority of the RoI's cross-template linkages pass through it. Its GET /contractual-arrangements/:id endpoint is implemented using Prisma's include operator to resolve all four nested relationships in a single database query: contractEntities (via contract_entities), contractProviders (via contract_providers), functionDependencies (via function_ict_dependencies), and entitiesUsingServices (via entities_using_services). Returning these nested arrays in a single response eliminates the N+1 query problem that would arise if the frontend fetched each relationship in a separate request — a pattern that is critical for contracts with many associated functions or entities (Fowler, 2002).
[20] The ContractEntitiesModule and ContractProvidersModule were implemented as separate NestJS modules with their own CRUD endpoints at /api/v1/contract-entities and /api/v1/contract-providers respectively. This separation — rather than embedding their logic inside ContractualArrangementsModule — reflects the single-responsibility principle: each module owns one table and one set of CRUD semantics, making the codebase easier to reason about and test independently (Martin, 2003). The frontend ContractualArrangements.tsx page integrates these modules through two sub-sections in the contract detail dialog: "Entities Party to This Contract" and "Additional Provider Signatories", each backed by real API persistence.
[21] 5.2.3 IctSupplyChain Module and Recursive Query
[22] The IctSupplyChainModule implements a recursive Common Table Expression (CTE) at GET /ict-supply-chain/chain/:contractId to traverse the parent_chain_id self-reference in ict_supply_chain and return the complete multi-tier subcontractor hierarchy for a given contract. This is the only endpoint in the system that uses a raw PostgreSQL query rather than a Prisma ORM call, because Prisma does not natively support recursive CTEs at the ORM level (Prisma, 2024). The raw query is parameterised — the contractId and tenantId values are passed as positional parameters $1 and $2 — ensuring that the departure from the ORM layer does not introduce SQL injection risk (OWASP, 2021).
[23] Role permissions for supply chain management are scoped so that both EDITOR and ADMIN role users can create, update, and delete supply chain entries, while READ operations remain accessible to all three roles. The @Roles(Role.ADMIN, Role.EDITOR) decorator is applied to the create, update, and delete controller methods. This operationalises DORA's expectation that entities maintain living, actively managed supply chain records, not static snapshots (European Commission, 2022, Art. 28(3)).
[24] 5.2.4 Settings Endpoints and User Self-Service
[25] Two user self-service endpoints were implemented in UsersController to support the Settings page: PATCH /api/v1/users/me/profile and PATCH /api/v1/users/me/password. The profile endpoint accepts { firstName, lastName, email } and updates the corresponding fields on the authenticated user's record. The password endpoint accepts { currentPassword, newPassword }, verifies currentPassword against the stored bcrypt hash using bcrypt.compare, and — if verification succeeds — hashes newPassword with ten salt rounds and persists the new hash. If verification fails, the endpoint returns a 401 Unauthorized, following OWASP guidance that password change endpoints should not distinguish between "wrong old password" and "user not found" in their error message (OWASP, 2021; NIST, 2018). Both endpoints are protected by JwtAuthGuard and all three roles can access them.
[26] 5.3 Database Implementation and Schema Management
[27] 5.3.1 Prisma Schema and Migration Approach
[28] The database schema is defined entirely in a single schema.prisma file, which serves as the canonical source of truth for the database structure, the Prisma client's TypeScript types, and the ORM's query API. This approach eliminates the class of bugs that arises from drift between the database schema and the application's data models — a common source of defects in systems where these layers are maintained independently (Fowler, 2002; Prisma, 2024).
[29] The current implementation uses prisma db push to synchronise the schema with the PostgreSQL database during development. prisma db push applies schema changes without generating migration files, which suits the iterative design-and-test cycle of a research prototype. However, it means there is no version-controlled migration history. For a production deployment, this would be replaced by prisma migrate dev / prisma migrate deploy, which generates and applies versioned SQL migration files and provides a full audit trail of schema evolution (Prisma, 2024). This is acknowledged as technical debt in Section 5.8.
[30] 5.3.2 Reference Data Seeding
[31] The prisma/seed.ts script initialises all nine reference tables from EBA-authoritative source data before any application data is created. The seed script uses Prisma's upsert operation — create or update keyed on the code value — so that it is idempotent: re-running the seeder on an already-populated database updates any changed display names without duplicating records or failing on constraint violations (Prisma, 2024). This idempotency property is important for a regulatory platform because EBA DPM code lists can be revised between regulatory periods, and the seeder must be able to synchronise the application's reference data with the latest published values without manual database intervention.
[32] The nine seeded reference tables correspond directly to the EBA's published code lists: country codes (ISO 3166-1 alpha-2, 249 values), currency codes (ISO 4217, 180 values), entity type codes (EBA DPM entity type taxonomy), criticality levels, reliance levels (EBA five-tier scale), data sensitivity levels (EBA four-tier classification), ICT service types, provider person types, and user roles. By seeding from these published lists rather than defining custom enumerations, the application's dropdown values are guaranteed to match the DPM taxonomy that the EBA's validation engine will apply to submitted RoI files (European Banking Authority, 2023a).
[33] 5.3.3 Domain Data Seeding and Test Fixtures
[34] After reference data is loaded, the seed script creates a synthetic demonstration tenant with a complete, realistic RoI dataset: three financial entities (one credit institution, one investment firm, one payment institution), five ICT providers with varied LEI formats and NACE codes, eight contractual arrangements at varying criticality levels, ten business functions with function-to-contract dependency mappings, six ICT service assessments, a supply chain with three tiers, and two exit strategy records. Several records contain deliberately injected data quality errors — a missing LEI on one entity, an invalid date format on one contract, a foreign-key orphan in one function dependency — which are designed to confirm that the validation engine detects the failure patterns that the ESA 2024 Dry Run identified as the most prevalent causes of submission failure (European Supervisory Authorities, 2024; Galati et al., 2022).
[35] This seeded dataset serves a dual purpose: it allows a new user to explore the platform's features without first entering data manually, and it provides a reproducible test baseline for validating the rule engine's detection capabilities. The injected errors are documented in a comment block within the seed script so that future developers understand their intentional nature and do not fix them inadvertently.
[36] 5.3.4 Multi-Tenancy Enforcement in Practice
[37] Every Prisma service method that queries domain data follows an identical tenant-scoping pattern: the tenantId is extracted from request.user, which is populated by the TenantIsolationMiddleware from the validated JWT payload, and it is passed as a mandatory where condition on every findMany, findFirst, update, and delete call. TypeScript's strict typing enforces that no where clause can be omitted: Prisma's generated client types make tenantId a required filter on all tables that carry the field, causing a compile-time error if a developer forgets to include it. This compile-time enforcement is an important property of the Prisma approach: the type system acts as a first line of defence against accidental cross-tenant queries, catching such omissions before the code is run (Prisma, 2024; Bezemer and Zaidman, 2010).
[38] Critically, application-layer enforcement is only one layer of the defence-in-depth strategy. PostgreSQL Row-Level Security (RLS) policies are applied to all 20 tenant-bearing tables via rls_policies.sql. These policies are keyed on the app.current_tenant_id session variable, which the TenantIsolationMiddleware sets on every request immediately after extracting the tenantId from the JWT. This means that even if an application-layer filtering omission were to occur, the PostgreSQL engine independently enforces tenant boundaries at the database level, consistent with the defence-in-depth architecture described in Chapter 4 (PostgreSQL Global Development Group, 2023; Rodero-Merino et al., 2010).
[39] 5.4 Security and Access Control Implementation
[40] 5.4.1 JWT Authentication Flow
[41] The authentication flow is implemented in AuthService and AuthController. When a user submits credentials to POST /api/v1/auth/login, AuthService.validateUser() queries for the user record by email, then calls bcrypt.compare(submittedPassword, user.passwordHash). On success, two tokens are issued by AuthService.generateTokens(): an access token with a 15-minute TTL, signed with the JWT_SECRET environment variable and containing the payload { sub: userId, email, tenantId, role }; and a refresh token consisting of 64 bytes of output from crypto.randomBytes(64).toString('hex'), which is hashed with bcrypt (ten rounds) and stored in users.refresh_token_hash.
[42] Both tokens are delivered as HttpOnly, SameSite: Strict cookies. The access token cookie is scoped to the path /api/v1/ so it is included in all API requests. The refresh token cookie is scoped narrowly to /api/v1/auth so it is only transmitted to the authentication endpoints. This path scoping means that if a cross-site scripting (XSS) vulnerability were exploited to trigger an arbitrary API request, the refresh token would not be included because the malicious request would target a non-auth path (OWASP, 2021; Anderson, 2020).
[43] Token refresh is handled by POST /api/v1/auth/refresh: the service retrieves the user record matching the sub claim, calls bcrypt.compare to verify the raw refresh token against the stored hash, and — if verification succeeds — issues a new token pair and overwrites refresh_token_hash with the new hash, immediately invalidating the previous refresh token (per-use rotation). A stolen refresh token can therefore only be used once before it becomes invalid (NIST, 2018).
[44] Password reset is implemented through a two-step flow: POST /api/v1/auth/forgot-password generates a cryptographically random 32-byte reset token, hashes it with SHA-256, stores the hash in users.reset_token with a one-hour expiry in users.reset_token_expires, and dispatches a password-reset email via the MailerModule. POST /api/v1/auth/reset-password verifies the submitted token against the stored hash, checks the expiry timestamp, and — on success — hashes the new password and clears the reset token fields, ensuring the token cannot be reused after expiry or a successful reset.

[46] 5.4.2 Role-Based Access Control Implementation
[47] The RBAC system is implemented through three components that work in sequence. The JwtAuthGuard extracts and validates the JWT from the access token cookie on every request, populating request.user with the token payload. The RolesGuard reads the @Roles(...) metadata set on the current controller or handler, retrieves request.user.role, and returns a 403 Forbidden response if the user's role is not in the permitted set. Within service methods, a service-level tenant check re-verifies that the record being accessed belongs to request.user.tenantId, providing defence-in-depth against any hypothetical guard bypass (Bass et al., 2021).
[48] The three roles are implemented as a TypeScript Role enum — ADMIN, ANALYST, EDITOR — referenced by both the @Roles() decorator and the users.role field in the Prisma schema, ensuring that the application code and the database schema use the same vocabulary. Custom decorators — @CurrentUser() and @TenantId() — extract user and tenant identity from the request object inside service methods, keeping services testable without requiring a full HTTP context.
[49] The practical capability boundaries are: ADMINs have full CRUD permissions across all domain data and all administrative functions within their tenant; EDITORs can create, update, and delete domain data records (including supply chain entries) but cannot manage users, trigger validation runs, or access administrative views; ANALYSTs can read all data and manage validation runs and issue state transitions but cannot create or modify domain data. These boundaries are enforced both by RolesGuard at the controller layer and by the route guard RoleGuard HOC in the frontend, which prevents role-inappropriate pages from rendering.
[50] 5.4.3 Audit Logging Implementation
[51] The AuditInterceptor is a globally registered NestJS interceptor that fires on every request matching a mutating HTTP method (POST, PATCH, PUT, DELETE). For PATCH, PUT, and DELETE requests, the interceptor pre-fetches the current record state through AuditFetchService before the controller handler is invoked. AuditFetchService maintains a mapping of HTTP route patterns to the corresponding Prisma model and primary key extraction logic, executing a targeted findUnique query against the correct table using the record ID from the route parameters. This pre-fetch captures the old_values snapshot; the interceptor then awaits the handler's completion, reads the updated record from the handler's response body as new_values, and writes the audit_logs entry.
[52] The sanitize() helper method strips credential-related fields — passwordHash, refresh_token_hash, reset_token, and reset_token_expires — from both old_values and new_values before persistence, ensuring that audit log entries never contain credential material. Both old_values and new_values are stored as PostgreSQL JSONB columns, preserving the full structural context of the change without requiring separate columns for each possible field (PostgreSQL Global Development Group, 2023). ADMIN-role users can view audit log entries through the /admin/audit-logs frontend page, which displays a filterable and sortable table of all logged operations with expandable before/after value panels for each entry.

[55] 5.5.1 Rule Seeding
[56] The prisma/seed-validation-rules.ts script (called from seed.ts during initialisation) loads 220 EBA validation rules into the validation_rules table, drawn directly from the EBA's published draft validation rules spreadsheet (EBA VR_01–VR_250). The rules are organised into seed batches by template — RT.01 rules are seeded first, followed by RT.02 through RT.08 in order, with RT.09 rules seeded last. This ordering ensures that any foreign-key-based rule (type fk_exists) is defined after the rules for the source template it references, which supports documentation clarity even though the execution engine resolves references at run time.
[57] Each seed entry specifies templateName, fieldName, ruleType, ruleValue, errorMessage, severity (ERROR, WARNING, or INFO), doraArticle (regulatory citation e.g. "Art.28(1)"), and isActive. The errorMessage field stores the human-readable description from the EBA's draft validation rules specification, making the validation_rules table self-documenting (European Banking Authority, 2024a). The executionOrder integer controls the sequence within each template, ensuring that required checks run before format checks — because a format rule applied to a null field would produce a misleading error rather than the correct "field is required" message.
[58] The engine supports ten rule types corresponding to distinct classes of EBA validation requirement: required, format, fk_exists, range, dropdown, cross_field, conditional, date_boundary, uniqueness, and aggregate. These are described in full in Chapter 4, Section 4.6.2. The 220-rule set provides complete mandatory-field coverage for RT.01 through RT.08 and partial coverage for RT.09, representing approximately 73% of the EBA's full published draft validation rule set (European Banking Authority, 2024a; Governatori et al., 2016).
[59] 5.5.2 Validation Run Execution
[60] When POST /api/v1/validation/run is received, ValidationService.executeRun() performs the following steps. First, it creates a validation_runs record with status RUNNING and records the start timestamp. Second, it queries validation_rules for all active rules, grouped by templateName. Third, for each template group, it retrieves the corresponding domain records from the relevant table filtered by tenantId. Fourth, it applies each rule to each record, generating validation_issues entries for each violation. Fifth, it computes the compliance score as (total_checks − violation_count) / total_checks and sets the run status to COMPLETED. Finally, it writes the category_summary JSONB breakdown to the validation_runs record, recording violation counts by template name, rule type, and severity.
[61] If an unexpected error occurs — for example, a malformed regex in a format rule's ruleValue — the error is caught by a top-level try/catch, the run status is set to FAILED, the error message is written to a run_error field on the validation_runs record, and any partially created validation_issues are preserved for diagnostic purposes. This graceful failure behaviour ensures that a single malformed rule does not silently suppress the entire run's output.

[63] 5.5.3 Pre-Flight Export Gate
[64] The pre-flight gate is implemented as a call from RoiExportService to ValidationService.checkPreflightStatus(tenantId: string) before any export generation begins. checkPreflightStatus queries the most recent completed validation_runs record for the tenant, then queries validation_issues for any issues with severity ERROR in status OPEN or FLAGGED. If any such issues exist, it returns a structured PreflightResult object with blocked: true and a list of the blocking issues. RoiExportService converts this into a 422 Unprocessable Entity response with the issue list embedded in the response body, allowing the frontend export dashboard to display specific issue descriptions and deep-links to the validation page. If no blocking issues exist, PreflightResult has blocked: false and export generation proceeds.
[65] This gate makes it structurally impossible to export a RoI containing unresolved ERROR-severity issues, directly addressing the most consequential failure mode identified in the ESA 2024 Dry Run — the submission of RoI files that fail mandatory EBA validation checks (European Supervisory Authorities, 2024).
[66] 5.6 RoI Export Implementation
[67] 5.6.1 Excel Export
[68] The Excel export is generated by RoiExportService.generateExcelWorkbook(tenantId: string) using ExcelJS version 4.4.0. The method creates a new Workbook instance, then for each RoI sub-template calls a dedicated addSheet() method that: queries the relevant domain table(s) filtered by tenantId; creates a worksheet named according to the EBA's template sheet naming convention; writes EBA-defined column headers as the first row with bold font and grey fill; and iterates the query results to add one row per record with values mapped to the corresponding EBA columns.
[69] Column mapping is implemented through a columnMap constant for each sub-template: an ordered array of objects specifying the EBA column header string, the Prisma result field path, and an optional transform function. The transform function handles cases such as boolean-to-Y/N conversion (for the "Critical or important function" field in RT.06.01), date-to-YYYY-MM-DD formatting, and code-list field resolution. This declarative mapping pattern means that adding a new column to an export requires only adding a new entry to the columnMap array, rather than modifying procedural generation logic.
[70] The GET /api/v1/roi/export endpoint streams the generated workbook directly to the HTTP response using ExcelJS's xlsx.write(response) method, with Content-Disposition: attachment; filename="RoI_[lei]_[YYYY-MM-DD].xlsx" and Content-Type: application/vnd.openxmlformats-officedocument.spreadsheetml.sheet headers.

[72] 5.6.2 XBRL OIM-CSV Export
[73] The XBRL OIM-CSV export is generated by RoiExportService.generateXbrlZip(tenantId: string). For each supported RoI sub-template, the method generates a CSV string using a custom serialiser that follows the EBA's OIM-CSV specification: no byte-order mark, UTF-8 encoding, first row as EBA-defined column headers, one data row per record, null values as empty fields, date values in YYYY-MM-DD format, and boolean values as true/false strings as required by the OIM schema (European Banking Authority, 2024b).
[74] The generated CSVs and a metadata.json file are assembled into a ZIP archive using the archiver library. The metadata.json contains { reportingDate, submissionPeriod, entityLei, schemaRef, generatedAt }, where schemaRef is a URI pointing to the EBA's published OIM-CSV taxonomy. As noted in Chapter 4, the schemaRef is a static value set at the time of development. If the EBA publishes a revised taxonomy version, schemaRef must be updated manually in the export configuration — this is acknowledged as a known limitation in Section 5.8. The CSV files are named according to the EBA convention — RT_01_01.csv, RT_02.csv, and so on — and the ZIP archive is named RoI_OIM_[lei]_[YYYY-MM-DD].zip.
[75] The GET /api/v1/roi/export/xbrl endpoint streams the ZIP archive with Content-Type: application/zip and Content-Disposition: attachment headers. The same ERROR-severity pre-flight gate applies as for the Excel export.

[81] 5.7.1 Application Shell and Authentication State (continued)
[82] On application load, an onRehydrateStorage callback silently attempts a token refresh via POST /api/v1/auth/refresh; if the refresh succeeds, the store is populated with fresh user data; if it fails — for example because the refresh token has expired or the tab was closed for longer than seven days — the user is redirected to the login page. This silent refresh pattern means that users who return to a live session within the refresh window never experience an unnecessary login prompt, which is important for a compliance workflow tool where users may leave the application open between data-entry sessions (OWASP, 2021).
[83] 5.7.2 Data Fetching and Mutation Pattern
[84] TanStack Query manages all server-state data throughout the application. Each domain feature area has a corresponding custom hook — useFinancialEntities, useContracts, useValidationRuns, and so on — that wraps a useQuery call with the appropriate endpoint URL, query key, and optional select transform. This pattern keeps data-fetching logic out of component bodies and centralises cache invalidation: when a mutation succeeds, the mutation's onSuccess callback calls queryClient.invalidateQueries([queryKey]), causing TanStack Query to refetch the affected data automatically without the component needing to manage refresh logic manually (Tanner Linsley, 2023).
[85] Mutations are implemented with TanStack Query's useMutation hook. Each form submission — creating a new entity, updating a contract, changing a password — calls the corresponding mutation function, which dispatches a PATCH or POST request to the backend. The mutation's onSuccess, onError, and onSettled callbacks show toast notifications via Shadcn's <Sonner> component, providing immediate user feedback without requiring page navigation. Error responses from the backend — which always follow the { statusCode, message, error } envelope described in Section 5.2.1 — are parsed to extract the human-readable message for display in the toast.
[86] 5.7.3 Core User Workflows
[87] The following describes the end-to-end workflow for each of the three primary user roles:
[88] EDITOR workflow — data entry and remediation. An EDITOR logs in and is directed to the Dashboard, which shows aggregate counts of entities, providers, contracts, functions, and open validation issues. The EDITOR navigates to /entities to create or update financial entity records, selecting entity type and country from server-populated dropdowns constrained to EBA DPM code-list values. From /contracts, the EDITOR creates a new contractual arrangement, selecting the provider from a dropdown populated by the IctProvidersModule API, and then adds entities party to the contract and provider signatories through the two sub-sections in the contract detail dialog. From /supply-chain, the EDITOR adds supply chain tiers by selecting contract, provider, subcontractor, and service type from dropdowns and entering the supply rank. When an ANALYST flags a validation issue to the EDITOR via the issue state machine, the EDITOR receives an in-app notification with a deep-link to the specific record. The EDITOR navigates to the record, corrects the data, and submits the issue for ANALYST review via the WAITING_APPROVAL state transition.

[94] 5.7.4 Notification and Comment Systems
[95] The in-app notification system is implemented through NotificationsModule, which creates notifications records linked to a target user ID when specific system events occur: an ANALYST flagging a validation issue to an EDITOR; an EDITOR submitting a fix; and an ANALYST resolving or rejecting a fix. The frontend polls GET /api/v1/notifications?unread=true on a 30-second interval using TanStack Query's refetchInterval option, and displays a badge on the notification bell icon in the top navigation bar when unread notifications exist. Notifications include a deep-link URL that navigates the user directly to the relevant record, eliminating the need to manually locate the affected data item (Butler and O'Brien, 2019).
[96] The comment system (CommentsModule) allows any authenticated user to attach timestamped comments to individual validation issues, contractual arrangement records, or ICT provider records. Comments are displayed in a chronological thread on the relevant detail page and are included in the audit log as COMMENT_ADDED entries, providing a contextual discussion history alongside the formal audit trail. This collaborative annotation capability addresses the finding in the RegTech literature that compliance workflows in SME organisations involve informal knowledge-sharing that needs to be captured within the system rather than dispersed across email threads (Butler and O'Brien, 2019; Arner et al., 2017).
[97] 5.8 Testing, Evaluation, and Known Limitations
[98] 5.8.1 Testing Approach
[99] The testing strategy for the research prototype is honest but limited. The platform has no dedicated automated test suite — no Jest unit tests, no Supertest integration tests, and no Playwright or Cypress end-to-end tests were written during the research implementation period. This is an acknowledged limitation of the Design Science Research approach as applied in a time-constrained master's research context: the research objective was to build a functional artefact that could be evaluated against its regulatory design requirements, not to achieve production-grade test coverage. Hevner et al. (2004) note that DSR artefacts are evaluated through demonstration against design requirements and expert review rather than exhaustive automated testing; this framing applies here.
[100] Functional correctness was verified through three approaches. First, the deliberately seeded data errors described in Section 5.3.3 were used as positive test cases: after each implementation iteration, a fresh validation run was executed and the results were checked against the expected error set to confirm that the engine detected the known violations at the correct severity levels. Second, the Swagger/OpenAPI interface at /api/docs was used to manually test all CRUD endpoints, verifying response structures and error handling against expected schemas. Third, the export outputs were manually compared against the EBA's published RoI template workbooks to verify column alignment, naming conventions, and data type handling (European Banking Authority, 2023a).

[105] 5.8.3 Known Limitations and Production Gaps
[106] The following limitations are acknowledged candidly as the gap between the prototype as built and a production-grade deployment:
[107] No automated test suite. The absence of Jest unit tests and integration tests means that regression defects introduced by future changes to the rule engine or export serialisers would not be automatically detected. The first production hardening step would be to add unit tests for the ten rule-type executor methods and integration tests for the pre-flight gate logic (Bass et al., 2021).
[108] No versioned migration history. The use of prisma db push means the schema cannot be incrementally rolled forward or backward. Production deployment requires migration to prisma migrate deploy, which generates a versioned SQL history compatible with zero-downtime deployment strategies (Prisma, 2024).
[109] No multi-factor authentication (MFA). The current authentication flow relies on password and refresh token only. For a financial compliance platform handling sensitive regulatory data, MFA — particularly TOTP or WebAuthn — would be required before production certification. NIST SP 800-63B Level 2 assurance requires at least two factors for systems handling sensitive personal or financial data (NIST, 2018).
[110] No encryption at rest. The PostgreSQL instance does not use tablespace-level encryption or application-layer field encryption for sensitive fields (such as contract terms or entity LEIs). In a production cloud deployment, transparent data encryption at the storage layer and, for the most sensitive fields, application-layer encryption using a key management service would be required (Anderson, 2020).
[111] Static XBRL schema reference. The schemaRef URI in the OIM-CSV metadata.json is a static value pointing to the EBA taxonomy version current at the time of development. If the EBA publishes a revised taxonomy, this must be manually updated in the export configuration (European Banking Authority, 2024b).
[112] Partial RT.09 coverage. The RT.09 sub-template for group-level entity-to-service links and concentration risk aggregation is the least mature component of the platform. Full implementation of RT.09 is the highest-priority feature development item for a production release.
[113] No automated tenant provisioning. Creating a new tenant currently requires a direct database insert by a system administrator. An automated onboarding flow — invitation email, guided setup wizard, initial seeding — would be required before the platform could operate as a self-service SaaS product.
[114] 5.9 Chapter Summary
[115] This chapter has traced the concrete implementation of the DORA SaaS platform from module structure through to known production gaps. The twenty-one NestJS modules implement the regulatory domain hierarchy as a clean, single-responsibility codebase in which the ContractualArrangementsModule serves as the relational hub for cross-template linkages. The shared-schema multi-tenancy model is enforced through a defence-in-depth strategy combining Prisma compile-time type enforcement, application-layer tenantId filtering via TenantIsolationMiddleware, and PostgreSQL Row-Level Security policies on all 20 tenant-bearing domain tables. The security architecture delivers JWT authentication with per-use refresh token rotation, role-based access control across three roles, rate limiting at ten requests per sixty seconds, and complete before/after audit logging via AuditInterceptor and AuditFetchService. The declarative rule-based validation engine seeds 220 EBA validation rules across ten rule types into the validation_rules table, executes them in a structured run lifecycle producing five-state validation_issues, and gates all export generation behind an ERROR-severity pre-flight check. The dual-path export subsystem produces fully conformant Excel workbooks and XBRL OIM-CSV ZIP archives for eleven of fourteen sub-templates, with RT.09 and RT.02 linking-table sections partially implemented. The frontend delivers role-differentiated workflows for EDITOR, ANALYST, and ADMIN users through a React 19 SPA with TanStack Query server-state management and SME-oriented constrained-dropdown UX patterns.
[116] The prototype's known limitations — the absence of an automated test suite, versioned migrations, MFA, encryption at rest, and full RT.09 coverage — are acknowledged explicitly and frame the scope within which Chapter 6's evaluation is conducted. Chapter 6 now proceeds to evaluate the artefact against the research objectives and the design requirements established in Chapter 3.
